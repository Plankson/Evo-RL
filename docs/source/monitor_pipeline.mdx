# Monitor Pipeline for `lerobot-human-inloop-record-monitor`

This document summarizes the dedicated detector-predictor recording flow added for the monitor pipeline.

## Entry point

The monitor path no longer runs through the generic `lerobot-human-inloop-record` loop. Use the dedicated entry point:

```bash
lerobot-human-inloop-record-monitor ...
```

The helper script is:

```bash
bash test_hil_record_monitor.sh
```

## Runtime workflow

The dedicated monitor recorder uses one shared observation source with two independent consumers:

1. The main control loop calls `robot.get_observation()` once per recorded frame, matching the normal HIL camera access pattern.
2. Each raw sample is published into an in-memory observation pool and detector queue.
3. The predictor samples from that pool, builds predictor-side inputs, and requests action chunks from `predictor_remote`.
4. The detector process receives the same raw observations from the shared source, builds detector-side inputs independently, and requests safety results from `detector_remote`.
5. Only frames actually consumed by the predictor/control loop are written into the final episode dataset.
6. Safety signals from predictor and detector are logged through the alarm path. Danger handling is still a TODO and does not interrupt execution.

The important constraint is: predictor and detector share the same raw observation source, but they are allowed to derive different final payloads before sending to their own remote endpoints.

## Required CLI arguments

Relative to a normal remote policy run, the monitor-specific arguments are:

```bash
--policy.type=remote_monitor
--policy.host=<server_host>
--policy.predictor_remote.port=<predictor_port>
--policy.detector_remote.port=<detector_port>
```

In practice you will usually also need:

```bash
--policy.predictor_remote.policy_name=<policy_name>
--policy.detector_remote.policy_name=<policy_name>
--policy.predictor_remote.chunk_size=<chunk_size>
--policy.predictor_remote.n_action_steps=<n_action_steps>
```

Optional monitor-recorder-only controls:

```bash
--observation_pool_size=<num_samples>
--detector_queue_size=<num_samples>
```

`--observation_poll_hz` is accepted for compatibility but ignored in the stable monitor path. Camera acquisition is intentionally kept in the main control loop because the RealSense backend uses background threads, and moving `robot.get_observation()` into an extra recorder-owned thread can destabilize the live camera pipeline.

If predictor and detector require different payload mappings, set them separately under their own nested prefixes. For example:

```bash
--policy.predictor_remote.image_key_map='{ ... }'
--policy.detector_remote.image_key_map='{ ... }'
--policy.predictor_remote.state_joints_indices='[...]'
--policy.detector_remote.state_joints_indices='[...]'
```

## Minimal command example

```bash
lerobot-human-inloop-record-monitor \
  --policy.type=remote_monitor \
  --policy.host=103.237.28.254 \
  --policy.predictor_remote.port=3336 \
  --policy.detector_remote.port=3337 \
  --policy.predictor_remote.policy_name=ace_policy \
  --policy.detector_remote.policy_name=ace_policy \
  --policy.predictor_remote.chunk_size=50 \
  --policy.predictor_remote.n_action_steps=24
```

The rest of the robot, teleop, dataset, and display arguments can stay aligned with the existing HIL setup.
